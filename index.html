2. Userspace analysis

When trying to analyze the difference in behavior between manual testing and xfstests, I noticed in tests/generic/472 that there are two different phases:

and the actual test body uses the xfstests helper:

So generic/472 has a probe phase and a test-body phase.  The probe uses the system mkswap(8)/swapon(8), where util-linux performs the userspace hole check.  The test body uses xfstests’ minimal src/mkswap/src/swapon; src/swapon does not perform the hole check and calls swapon(2) directly.

As mentioned in the previous section, there is a delay in metadata updates. Consequently, I tried the cache=none mount option to bypass client-side caching. However, the 10-page (40KB) probe still failed the util-linux swapon(8) hole check:

Next, I increased the swapfile size and it unexpectedly passed the hole check. This is likely due to server-side XFS speculative preallocation (EOFblocks). For larger files, XFS pre-allocates extra blocks, which masks the metadata lag by keeping the reported block count higher than the actual file size:

After this change, the probe passed, but generic/472 still failed later in the

3. Kernel-side analysis

Issue Investigation

CIFS cifs_swap_activate() has a similar allocation check based on inode metadata:

With a debug printk before the check, successful and failing activations can be
separated clearly. For example, after the swapfile was enlarged to 20 pages, the
probe activation succeeded:

Later activations from the real generic/472 body reached the kernel and failed:

Attempt 1: revalidate before the cifs_swap_activate() hole check

The userspace investigation already showed that CIFS allocation metadata can lag behind EOF in this workload. Therefore, the kernel-side hole check in cifs_swap_activate() should not be performed using stale inode->i_blocks.

My first attempt was to force revalidation immediately before reading i_blocks and i_size in cifs_swap_activate()：

By combining this change with a cache=none mount and an xfstests-dev workaround that increased the require_scratch_swapfile() probe swapfile size to at least 20 pages, almost all of the CIFS swap-related xfstests successfully passed in my environment.

Attempt 2: make actimeo=0 override CIFS_CACHE_READ

I also ran the same tests over NFSv4.2 with the noac mount option. Almost all tests passed without increasing the _require_scratch_swapfile() probe swapfile size.

This led me to compare how NFS/noac and CIFS/actimeo=0 handle attribute revalidation. In the NFS client, noac disables the attribute cache and nfs_getattr() can force a revalidation through:

In contrast, tracing the CIFS path showed that actimeo=0 did not always force a full attribute revalidation.

ftrace showed that cifs_revalidate_dentry_attr() was entered, but
cifs_dentry_needs_reval() returned false before reaching the acregmax == 0
check, because CIFS_CACHE_READ was true for the inode:

cifs_dentry_needs_reval: needs_reval false: cache_read ... acregmax=0

The relevant ordering was:

Even if cifs_dentry_needs_reval() returned true, the revalidation could still stop in cifs_get_inode_info(): is_inode_cache_good() did not check acregmax == 0.

So I changed CIFS to make disabled attribute caching take precedence over
CIFS_CACHE_READ in both cifs_dentry_needs_reval() and is_inode_cache_good().

smb: client: make actimeo=0 override CIFS_CACHE_READ

With this patch and cache=none,actimeo=0, the tests no longer needed the
xfstests userspace workaround of increasing the _require_scratch_swapfile()
probe size, and almost all CIFS swap-related xfstests passed in my environment.

4. Result summary

After the following changes, all originally reported tests passed except generic/495
and generic/496.

generic/495 was a separate server-reported block-size issue. The test requires
the swapfile allocation unit size to match the page size. Setting the Samba global
block size to 4096 made the reported value match the 4 KiB page size, and
generic/495 passed:

generic/496
The reason is that generic/496 creates a swapfile with fallocate. CIFS currently extends the file mainly via SMB SET_EOF. Without strict allocate, Samba may extend the logical file size without fully allocating backing blocks, so CIFS later sees i_blocks < i_size and cifs_swap_activate() rejects the file as having holes.

With strict allocate = yes, Samba actually preallocates the backing space when the file is extended.

And then the test case generic/496 passed:

5. Conclusion

These failure was caused by stale CIFS allocation metadata: the hole checks used cached i_blocks/st_blocks that lagged behind EOF, so the swapfile was incorrectly rejected as sparse.

All originally reported tests passed with the following changes except generic/495 and generic/496:

The userspace mount whith option cache=none,cifsacl,actimeo=0 .
